# llmNote 笔记 Worktree + PR 工作流

你负责在 `llmNote` 仓库里完成一整套笔记交付流程：创建 worktree、修改或新增笔记、构建验证、提交 commit、推送分支并创建 PR。

## 适用范围

- 目标仓库固定为：`/Users/bytedance/codes/llmNote`
- 适合笔记型任务：新增一篇主题笔记、补充现有笔记、改写某篇学习笔记并提交 PR
- 用户通常只会给出一个主题或一句任务描述，你需要自动补齐分支名、slug、提交信息与 PR 流程

## 默认约定

- 主分支默认使用 `main`
- 创建 worktree 时一律以最新的远端 `origin/main` 为基准，而不是依赖本地可能落后的 `main`
- worktree 路径默认使用：`/Users/bytedance/codes/.worktrees/llmNote/<branch>`
- 新笔记路径默认使用：`src/pages/notes/<slug>.astro`
- 笔记需要导出 `meta`
- 首页通常不需要手动修改，因为 `src/pages/index.astro` 会自动收集 `src/pages/notes/*.astro`
- 构建命令固定使用：`npm --prefix "<worktree>" run build`

## 笔记内容规范

### 写作目标

- 默认目标不是“写一篇简短介绍”，而是交付一篇像技术 blog 一样层层展开、细节充分、便于连续阅读的长文笔记。
- 除非用户明确要求写成短文，否则优先写成“可独立阅读的完整文章”，而不是提纲、摘要或扩写版 bullet list。
- 对每个核心概念都要回答至少这几类问题：它是什么、为什么重要、它如何工作、关键组成部分是什么、它与相近概念有何区别、在真实系统里如何使用、常见误区和局限是什么。
- 如果主题本身涉及系统、框架、论文、工具链或方法论，优先写出“背景 + 机制 + 结构 + 流程 + 例子 + 边界 + 实践建议”的完整闭环。

### 调研深度要求

- 在真正写正文前，必须先做多来源调研，不能只依赖单一网页、单篇博客或用户给的一条链接直接改写。
- 默认至少收集并交叉阅读 3 类来源；如果主题较复杂，优先达到 4 到 6 个有效来源。
- 来源优先级：一手资料（论文原文、官方文档、官方仓库、官方 benchmark、源码） > 高质量二手资料（权威技术博客、课程、深度解读） > 一般性总结。
- 如果用户只给了一个参考链接，也必须继续补充相关的一手或高质量补充来源，用来核对术语、结构和细节。
- 调研时不要只摘结论，要提取可用于正文展开的内容：定义方式、设计动机、关键模块、参数含义、图示解释、实验设置、实现细节、版本差异、社区常见误解。
- 写作时要把不同来源的信息真正融合进正文，用于解释和展开，而不是在末尾简单罗列链接。

### 通用笔记要求

- 笔记要有研究友好的深度，包含关键公式、参数定义与必要假设。
- 内容结构优先遵循：背景 -> 核心概念 -> 公式/推导 -> 训练或推理流程 -> 实践建议。
- 图示应嵌入到对应讲解段落附近，不要做成分离式图片堆叠。
- 优先使用能表达真实关系的图：参数关系、层次结构、模块连接、数据流，而不是泛化流程图。
- 小节内容要扎实，优先用子标题组织清楚，避免只有摘要式短段落。
- 公式必须居中、清晰、大小与正文协调，不能只是随意塞一段 LaTeX。
- 默认写成多章节长文，优先包含 5 个及以上有信息增量的一级或二级章节，而不是 2 到 3 个大段快速收束。
- 每个关键章节都要展开讲“机制”和“细节”，避免只有定义句和结论句；如果某一节只有 1 小段摘要，应继续补充例子、拆解或对比。
- 尽量加入具体例子、运行流程、模块拆分、输入输出说明、失败模式或实践经验，让内容像深度博客而不是读书卡片。
- 对容易混淆的概念，优先加对比段落，例如“它不是什么”“它和相邻概念的边界是什么”。
- 如果主题有可解释的系统流程，优先拆成多个阶段逐段讲，而不是一句话概括整个 pipeline。

### 论文型笔记附加要求

- 如果来源是论文，必须先读取并遵循仓库中的 `NOTE_STYLE_REQUIREMENTS.md`。
- 整体结构优先遵循：Introduction -> Background/Motivation -> Method/System -> Experiments/Evaluation -> Discussion -> Conclusion。
- 段落要完整，保持易懂但专业的讲解风格。
- 在关键章节加入原文引用和来源链接，优先放在 Abstract、Intro、Results 等位置。
- 尽量收集并使用论文原图，在对应章节嵌入。
- 原图文件下载到 `public/notes/<slug>/`，并在页面中本地引用。
- 图片说明文字要具体，图号尽量和原论文保持一致。
- 文章末尾保留 `Original Figures` 区域，方便快速浏览原图。
- 迭代时优先追加内容，不要无故删除已经存在的高价值段落。

### 参考资料要求

- 优先使用一手资料：论文原文、官方文档、官方仓库、官方 benchmark 或官方实现。
- 如果引用博客、课程或二手解读，只能作为补充，不能替代核心一手来源。
- 参考资料部分至少给出最关键的原始来源链接；如果是论文型笔记，原论文链接必须出现。
- 引用和参考资料要服务于正文，而不是仅在末尾堆链接。
- 如果正文中的关键解释来自某个具体来源，应在相关章节附近明确指出，而不是把所有出处都丢到最后。
- 对多来源结论，优先写出“为什么这样理解”：哪些内容来自官方定义，哪些来自实现细节，哪些来自二手材料的补充解释。

### 章节与篇幅要求

- 默认把文章写成完整的 blog 结构：开篇背景、问题定义、核心机制、关键结构或公式、运行/训练/推理流程、例子或实践分析、局限与总结。
- 如果主题较复杂，允许并鼓励写得更长；不要为了“像笔记”而主动压缩成短文。
- 除非主题本身非常小，否则不要交付“通篇都很短”的文章；正文应足够长，能支撑连续阅读和系统理解。
- 单个章节应承载明确的信息增量，不要把多个主题压缩到同一节的几个 bullet 里。
- 可以使用 bullet，但 bullet 只能辅助归纳，正文主体仍应是完整段落和分节讲解。
- 在落笔前先规划章节大纲；如果发现内容仍然偏摘要式，先补章节再补细节，不要直接进入提交阶段。

## 命名规则

- 如果是新增笔记，优先使用分支名：`add-<slug>-note`
- `slug` 使用英文 kebab-case，例如：`harness`、`rlhf-judge`
- PR 标题优先使用：`Add <Topic> note`
- commit message 优先使用简短、祈使句风格，例如：`Add harness introduction note`

## 执行步骤

1. 先检查仓库结构与已有笔记，确定是新增还是修改现有页面。
2. 为主题生成合适的 `slug`、分支名与 worktree 路径。
3. 在真正写笔记前，必须先读取仓库里的 `AGENTS.md`；如果属于论文型笔记，还必须读取 `NOTE_STYLE_REQUIREMENTS.md`。
4. 在真正开始写正文前，必须先完成一轮多来源调研，并整理出计划写入文章的章节结构与核心问题。
5. 在主仓库上执行 `git fetch origin --prune`，确认 `origin/main` 已更新到最新，再基于最新的 `origin/main` 创建新的 worktree。
6. 如果新 worktree 没有依赖，先执行 `npm --prefix "<worktree>" install`。
7. 写作时必须显式对照“笔记内容规范”逐项落实，而不是只凭印象生成；优先先搭出章节骨架，再把每节写厚。
8. 笔记写入或修改完成后，先做一次内容自检，再执行 `npm --prefix "<worktree>" run build` 验证。
9. 提交前查看 `git status`、`git diff` 与最近的提交风格。
10. 使用 `git add`、`git commit`、`git push -u origin <branch>` 完成提交与推送。
11. 使用 `gh pr create` 创建 PR，并返回 PR 链接、分支名、worktree 路径与构建结果。

## 强制执行顺序

- 不允许跳过“读取规范 -> 多来源调研 -> 规划章节 -> 写作 -> 内容自检 -> 构建验证 -> git 提交 -> PR 创建”这条链路。
- 不允许在未做内容自检时直接提交。
- 不允许在构建失败时直接提交 PR，除非用户明确要求忽略验证。
- 如果发现输出内容与规范不一致，必须先修正内容，再继续后续步骤。
- 如果文章明显偏短、偏总结、偏提纲化，必须先补调研和补章节，再继续后续步骤。

## 写作前必做检查

- 判断这是“普通主题笔记”还是“论文型笔记”。
- 读取 `AGENTS.md` 中的通用写作要求。
- 如果是论文型笔记，额外读取 `NOTE_STYLE_REQUIREMENTS.md`。
- 确认本次写作计划使用哪些来源，是否覆盖至少一手资料与补充解释来源。
- 先列出文章的大纲，确认不是只有几个总结性章节，而是能支撑长文展开的结构。
- 判断是否需要新增本地图片目录 `public/notes/<slug>/`。
- 判断是否已有相近笔记可以复用术语、结构或图示风格。

## 提交前内容自检清单

在 `git add` 之前，必须逐项自检：

- 是否包含核心概念解释，而不是只有结论性摘要。
- 是否确实做了多来源调研，而不是基本只围绕单一链接改写。
- 是否把调研结果写进了正文展开，而不是只在末尾列出几个参考链接。
- 是否已经形成多章节结构，且章节之间有清晰递进，而不是几段大而空的概述。
- 文章是否明显过短；如果读起来更像摘要卡片而不是 blog，应继续补充。
- 每个关键章节是否讲清了原理、流程、结构、例子、边界或实践点中的至少若干项，而不是只有定义和结论。
- 如果主题需要公式，是否真的写出了关键公式，并解释了参数与含义。
- 公式是否居中、可读、样式与正文协调。
- 图片是否嵌入到了相关段落附近，而不是孤立堆在一起。
- 图片是否在解释真实关系，而不是泛泛流程图。
- 参考资料是否优先采用了一手来源。
- 对引用的关键来源，是否解释了它们各自补充了什么信息，而不是只“引用一下”。
- 如果是论文型笔记，是否加入了原文引用和原始来源链接。
- 如果是论文型笔记，原图是否放到了 `public/notes/<slug>/` 并本地引用。
- 如果是论文型笔记，末尾是否保留了 `Original Figures` 区域。
- 章节结构是否清楚，段落是否完整，不是只有碎片化 bullet。
- `meta` 是否完整，编号、标题、描述、标签、阅读时长是否合理。

如果以上任一项不满足，优先修正文稿，不要直接进入 git / PR 阶段。

## 强制要求

- 不要只给命令示例，直接执行。
- 新 worktree 必须来自最新的 `origin/main`；如果发现当前 worktree 不是从最新 `origin/main` 切出，优先重建或明确修正基线。
- 遇到 `astro: command not found` 时，优先判断是 worktree 未安装依赖，并主动执行 `npm install` 后重试。
- 创建 PR 时，`--body` 一律使用 `HEREDOC` 风格包裹，避免 shell 误解释反引号或路径。
- 默认按“深度技术 blog”标准写作，而不是按“简短学习笔记”标准收缩内容。
- 如果主题允许展开，宁可写得更完整，也不要只交付一个安全但过短的版本。
- 除非用户明确要求，否则不要把主体内容写成大段 bullet，总结只能放在章节末尾或文末。
- 对关键概念优先写出细节解释、来源依据和结构拆解，而不是只下定义。
- 必须在最终结果中明确说明：是否完成了内容自检，以及构建是否通过。
- 在最终回复里必须包含：
  - worktree 绝对路径
  - 修改的主要文件路径
  - 本次采用的核心来源类别或代表性来源
  - 内容自检是否完成
  - 构建命令是否通过
  - commit hash
  - PR 链接

## PR 正文模板

创建 PR 时，正文优先使用下面结构：

```text
## Summary
- add or update the target note under src/pages/notes/
- summarize the main conceptual or structural changes
- ensure the site builds successfully

## Testing
- npm --prefix "<worktree>" run build
```

## 提交命令要求

提交 commit message 时，优先使用这种安全格式：

```bash
git -C "$WORKTREE" commit -m "$(cat <<'EOF'
Add <topic> note

Brief explanation of the note change.
EOF
)"
```

## PR 命令要求

创建 PR 时，优先使用这种安全格式：

```bash
gh pr create \
  --repo TianCai19/llmNote \
  --base main \
  --head "$BRANCH" \
  --title "$PR_TITLE" \
  --body "$(cat <<'EOF'
## Summary
- add or update the target note under src/pages/notes/
- summarize the main conceptual or structural changes
- ensure the site builds successfully

## Testing
- npm --prefix "<worktree>" run build
EOF
)"
```

## 质量标准

- 笔记结构清晰，符合当前仓库的 Astro 页面风格
- 与主题相近的既有笔记保持术语、标题层级和视觉布局一致
- 写作时必须遵守本 Skill 中的“笔记内容规范”，尤其是公式、图片、本地资源与引用来源要求
- 如果是论文型笔记，必须显式应用 `NOTE_STYLE_REQUIREMENTS.md` 中的结构与媒体规则
- 不做无关修改
- 如果首页可以自动收录，不要额外改首页
- 只有在用户明确要求时才做额外的 git 操作（如 merge、close PR、删除 worktree）

## 结果汇报模板

完成后，用简洁中文汇报：

- 已创建的 worktree 路径
- 新建或修改的笔记文件
- 本次采用的核心来源类别或代表性来源
- 内容自检结果
- 构建验证命令与结果
- commit hash
- PR 链接

如果任务信息不足以确定主题 slug，可以根据主题自动推断一个简洁合理的英文 slug，然后继续执行。
